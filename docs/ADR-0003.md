# ADR-0003

## Title

UKA v1.0 Runtime Resolution Capability Scope

—

## Status

ACCEPTED  
LOCKED

—

## Date

July 2026

—

# Context

ADR-0001 established UKA as the Knowledge Engine of AI Prompt Studio.

ADR-0002 established the Knowledge Runtime Package as the official runtime contract between the UKA publishing system and AI Prompt Studio.

During Fase 7 — Runtime Consumer Integration and Fase 8 — Knowledge Resolution / Intelligence, the complete knowledge pipeline was integrated and audited end to end:

```text
UKA Official Master Baseline
        ↓
Knowledge Importer
        ↓
Internal Knowledge Contract
        ↓
Knowledge Publisher
        ↓
Knowledge Runtime Package
        ↓
AI Prompt Studio Runtime Consumer
        ↓
Knowledge Resolution
        ↓
Prompt Generator
```

The audit established that the target knowledge architecture and the capabilities currently published by UKA v1.0 are not identical.

The runtime contract is designed to support multiple knowledge registries and future capabilities.

However, AI Prompt Studio must resolve knowledge only against capabilities that are actually available through the current published runtime data.

Application code must not fabricate missing knowledge in order to simulate future architecture.

—

# Decision

For UKA v1.0, AI Prompt Studio shall perform runtime Knowledge Resolution using only the knowledge capabilities currently published and available for active resolution:

```text
Canonical Outputs
        +
Master Domains
```

The active UKA v1.0 resolution flow is:

```text
Creator Idea
        ↓
Canonical Resolution
        ↓
Parent / Master Domain Resolution
        ↓
Knowledge Context
        ↓
Prompt Generator
        ↓
Generated Prompt
```

The following capabilities are not part of the active UKA v1.0 resolution scope:

- Intent
- Alias
- Relationship
- Structured Context

These capabilities must not be fabricated, inferred as authoritative knowledge, or hardcoded in the AI Prompt Studio application layer.

They may be integrated only when they are officially published by UKA through the governed knowledge pipeline and Knowledge Runtime Package.

—

# Runtime Contract Compatibility

This decision does not remove or invalidate the forward-compatible runtime model established by ADR-0002.

The Knowledge Runtime Package may retain registry structures that are currently empty or not active for resolution.

For example:

```text
registries
├── intents
├── domains
├── canonicals
└── relationships
```

The existence of a registry in the runtime contract does not mean that the registry currently contains an active published dataset.

For UKA v1.0:

```text
Domain Registry
ACTIVE

Canonical Registry
ACTIVE

Intent Registry
FUTURE / NOT ACTIVE FOR RESOLUTION

Relationship Registry
FUTURE / NOT ACTIVE FOR RESOLUTION
```

The runtime contract remains forward-compatible so future UKA capabilities can be integrated without requiring application-layer knowledge fabrication.

—

# Canonical Resolution Decision

Canonical Resolution is the active Knowledge Resolution capability for UKA v1.0.

The resolver operates against published canonical records and their associated parent domain identifiers.

The active runtime relationship is:

```text
Canonical
        ↓
parentDomainId
        ↓
Master Domain
```

The resolver does not use unpublished:

- Intent mappings
- Alias mappings
- Invented synonyms
- Hidden category mappings
- Fabricated relationships
- Structured Context
- External semantic knowledge presented as UKA knowledge

—

# Resolution Signals

For UKA v1.0, valid resolution signals are limited to information available through the published runtime and creator input.

The active resolution process is:

```text
Creator Idea
        ↓
Text Normalization
        ↓
Exact Canonical Phrase Match
        ↓
Canonical Token Matching
        ↓
Match Quality Evaluation
        ↓
Candidate Ranking
        ↓
Confidence Gate
        ↓
Best Canonical Match
        ↓
Parent / Master Domain Resolution
```

The resolver may return `null` when no candidate is sufficiently strong.

A null resolution is valid behavior.

No Knowledge Context is preferable to incorrect Knowledge Context.

—

# Scoring and Ranking

The UKA v1.0 resolver uses deterministic scoring and ranking.

The locked initial scoring rules are:

```text
Exact normalized idea match
Score: 100

Exact canonical phrase match
Score: 90

Full token match
Base score: 70 + specificity bonus

Partial token match
Eligible only when:
- matched tokens >= 2
- token coverage >= 60%

Initial confidence threshold
Score >= 60
```

The confidence threshold is an application resolution parameter and is not represented as authoritative UKA knowledge.

It may be calibrated through future testing without changing the UKA source of truth.

—

# Single-Token Protection

Single-token canonicals require conservative handling.

A single-token canonical may be accepted only through exact whole-token or phrase matching.

The resolver must not use fuzzy or partial expansion to force a single-token match.

Whole normalized tokens are compared directly.

The resolver does not automatically infer unpublished linguistic equivalences such as:

```text
plan = planning
strategy = strategic
market = marketing
```

Such equivalences require governed knowledge capabilities if they are introduced in the future.

—

# Candidate Ranking

When multiple candidates are valid, the resolver ranks them deterministically.

The locked tie-breaking order is:

```text
1. Higher match type
2. Higher token coverage
3. More matched tokens
4. Greater canonical specificity
5. Stable canonical ID
```

The resolver must not depend on accidental array order to determine the winning canonical.

—

# Confidence Gating

The resolver must be allowed to return:

```text
canonical: null
domain: null
```

when no candidate satisfies the required confidence rules.

This protects the application from:

- Generic-input false positives
- Unsupported knowledge assumptions
- Weak partial matches
- Knowledge coverage gaps
- Fabricated resolution

The absence of Knowledge Context is intentional when resolution confidence is insufficient.

—

# Runtime Consumer Implementation

The active application implementation is:

```text
knowledge-runtime.json
        ↓
runtime-loader.js
        ↓
runtime-resolver.js
        ↓
generator.js
        ↓
Generated Prompt
```

`runtime-loader.js` loads and exposes the published runtime package.

`runtime-resolver.js` performs Canonical Resolution and Parent / Master Domain Resolution.

`generator.js` consumes the resolver result and adds `KNOWLEDGE CONTEXT` only when resolution succeeds.

The existing public resolver API is preserved:

```text
resolveCanonical
resolveDomain
resolveRuntimeKnowledge
```

—

# Validation

The scoring and ranking design was first evaluated through simulation against the actual UKA v1.0 runtime dataset containing 972 canonical records.

The simulation confirmed:

- Exact phrase matching
- Reordered full-token matching
- Generic-input rejection
- False-positive protection
- Deterministic null fallback

Cases involving unavailable canonical concepts were treated as knowledge coverage gaps rather than resolver failures.

The resolver must not fabricate missing canonical knowledge.

—

# Consequences

## Positive

- AI Prompt Studio uses only knowledge actually published by UKA.
- The application does not fabricate missing knowledge capabilities.
- Canonical Resolution works without requiring unfinished future registries.
- False positives are reduced through scoring and confidence gating.
- Unsupported or ambiguous input can safely return null.
- The runtime contract remains forward-compatible.
- Future UKA capabilities can be integrated through the governed publishing pipeline.

## Trade-offs

- Resolution quality is limited by current UKA knowledge coverage.
- Concepts not represented by published canonical data may return null.
- Intent, Alias, Relationship, and Structured Context capabilities are unavailable until officially published.
- Current deterministic token matching does not provide unrestricted semantic synonym inference.
- Resolution thresholds may require future calibration through broader testing.

—

# Architecture Boundaries

## UKA Owns

- Authoritative knowledge
- Master Domains
- Canonical Outputs
- Future Intent knowledge
- Future Alias knowledge
- Future Relationship knowledge
- Future Structured Context
- Knowledge governance

## Knowledge Publishing Pipeline Owns

- Source ingestion
- Knowledge normalization
- Runtime transformation
- Runtime validation
- Runtime package generation

## AI Prompt Studio Owns

- Runtime loading
- Resolution against published capabilities
- Candidate scoring
- Candidate ranking
- Confidence gating
- Prompt generation
- Creator experience

AI Prompt Studio does not own authoritative UKA knowledge.

—

# Future Capability Integration

When UKA officially publishes additional knowledge capabilities, AI Prompt Studio may extend the resolution flow.

A future resolution architecture may include:

```text
Creator Idea
        ↓
Intent Resolution
        ↓
Alias Resolution
        ↓
Canonical Resolution
        ↓
Relationship Resolution
        ↓
Structured Context Resolution
        ↓
Prompt Generator
```

This future architecture is not the active UKA v1.0 runtime capability.

Future capabilities must enter the application through:

```text
UKA
        ↓
Knowledge Importer
        ↓
Internal Knowledge Contract
        ↓
Knowledge Publisher
        ↓
Knowledge Runtime Package
        ↓
AI Prompt Studio
```

Application-layer hardcoding must not be used to bypass this pipeline.

# Final Decision

For UKA v1.0, AI Prompt Studio performs Knowledge Resolution using **Canonical Outputs and Master Domains**.

Intent, Alias, Relationship, and Structured Context are future capabilities and must not be fabricated or hardcoded in the application layer.

Canonical Resolution uses deterministic candidate evaluation, scoring, ranking, confidence gating, and null fallback behavior.

The current implementation has passed 5/5 live application tests.

The Knowledge Runtime Package remains the official forward-compatible runtime contract.

This Architecture Decision is **ACCEPTED** and **LOCKED**.

Future architectural changes to the active runtime resolution capability scope must be introduced through a new Architecture Decision Record.
