# ADR-0002

## Title

Knowledge Runtime Package Architecture

—

## Status

ACCEPTED

LOCKED

—

## Date

July 2026

—

# Context

UKA Official Master Baseline is the authoritative knowledge source for AI Prompt Studio.

However, the UKA workbook contains authoring data, governance records, audit history, and documentation that are not suitable for runtime use.

AI Prompt Studio requires a lightweight, versioned, and immutable runtime representation optimized for prompt generation.

—

# Decision

AI Prompt Studio adopts the **Knowledge Runtime Package** as its official runtime contract.

The Knowledge Runtime Package is generated exclusively by the Publisher subsystem.

AI Prompt Studio never consumes the UKA workbook directly.

—

# Runtime Architecture

```
UKA Official Master Baseline
        │
        ▼
Publisher
        │
        ▼
Knowledge Runtime Package
        │
        ▼
AI Prompt Studio
```

—

# Runtime Model

The Knowledge Runtime Package is registry-based.

It contains independent registries.

- Intent Registry
- Domain Registry
- Canonical Registry
- Relationship Registry
- Runtime Metadata

Registries are connected through immutable IDs.

—

# Registry Principles

The runtime model follows these principles.

- ID-first
- Registry-based
- Immutable
- Versioned
- Platform-independent

Relationships are represented through IDs rather than nested objects whenever possible.

—

# Publisher Responsibilities

The Publisher is responsible for:

- Registry validation
- Relationship resolution
- Runtime transformation
- Package generation
- Package validation
- Versioning

The Publisher is the only subsystem permitted to generate runtime packages.

—

# AI Prompt Studio Responsibilities

AI Prompt Studio is responsible for:

- Loading runtime packages
- Intent resolution
- Prompt generation
- Creator experience

AI Prompt Studio never modifies runtime packages.

—

# Benefits

This architecture provides:

- Single Source of Truth
- Reduced duplication
- Smaller runtime packages
- Better scalability
- Faster loading
- Easier caching
- Platform independence

—

# Consequences

## Positive

- UKA evolves independently from AI Prompt Studio.
- Runtime packages remain lightweight.
- Publisher becomes a reusable subsystem.
- Runtime data is deterministic and versioned.

## Trade-offs

- Publisher becomes a required build step.
- Runtime packages must be regenerated whenever UKA changes.
- Schema compatibility must be maintained across releases.
